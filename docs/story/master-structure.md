# Мастер-структура Don’t Look to the Sky

**Статус:** главный структурный документ проекта.
**Утверждён автором:** 21 июля 2026 (хребет, состав маршрутов, судьба старых рутов).
**Приоритет:** при противоречии с любым другим файлом планов источник истины — этот документ. Расхождение устраняется правкой обоих файлов, а не молчаливым накоплением версий.

Производные документы (остаются как детали, не как норма):
- `common-week-plan.md` — решения и состав общей недели;
- `story-structure.md` — мир, тема, обещания истории;
- `routes/*.md` — жанровые контракты маршрутов;
- `../bible/*` — канон персонажей и космологии.

## 1. Хребет

```text
ДНИ 1–7      ДЕНЬ 8        ДНИ 9–12           ДНИ 13–21
общая        три версии    переплетение       ведущая линия
неделя       дня по        общего календаря   в едином мире
C00–C06      приоритету    и уникальных       дни всё сильнее
(общие)      (М / В / N)   блоков             расходятся
                              │
                    после дня 12 — скрытый resolver
                              │
                    утро дня 21 — сверка статуса:
                    исполняется результат траектории
```

Пять твёрдых правил:

1. Общая неделя не заканчивается выбором маршрута. В конце дня 7 Александр выбирает **занятие дня 8**, а не судьбу.
2. Дни 8–12: один календарь, общие объективные события по датам, уникальные блоки текущего приоритета; расхождение нарастает постепенно, а не рублем.
3. Resolver после дня 12 — скрытый, по поступкам и обязательствам, без кнопки «выбери девушку».
4. Дни 13–21: ведущая линия, но мир не распадается — работа, квартира, срок и все ключевые персонажи продолжают жить в каждой линии.
5. Позднее «перебегание» в чужой маршрут после resolver — повреждение текущей линии, а не спасение другим финалом.

## 2. Слоты дня и типы сцен

День состоит из четырёх слотов: **утро → смена/дела → вечер → ночь**. Сцена занимает слот или его часть.

| Тип | Обозначение | Правило хранения | Правило написания |
|---|---|---|---|
| Общая | **О** | Один экземпляр текста на проект | Проигрывается всем; чужие мотивы не допускаются, вставки через переменные |
| Вариативная | **В** | Один файл, ветвящиеся фрагменты по условиям | Костяк общий; участники, знания и продолжения зависят от прожитого |
| Уникальная | **У** | Отдельный блок только своего приоритета/маршрута | Полная свобода линии; обязана уважать объективный календарь |

Запрет: **О-сцены не копируются под маршруты.** Разница между линиями создаётся В-вставками и У-блоками, а не дублированием текста — иначе возвращаемся к бардаку «переписывать всё по сто раз».

## 4. Контракт пересечений

Кто живёт в чужой линии и как:

- **Работа (Дмитрий, смены, депо).** Объективный слой мира: в каждой линии есть рабочие даты. Смена пишется один раз `[О/В]`, меняются доступ и последствия.
- **Лена.** Земной якорь во всех линиях, дозированно. В N — главная земная линия участника поиска; в М и В — как минимум два касания (одно рабочее, одно личное).
- **Мираэль.** В линии В: самостоятельно участвует в решении своего статуса, без молчаливой статуэтки. В линии N: соискатель ответа с открытым личным финалом.
- **Валерия.** В линии М: давление стандартного договора и срока по датам. В линии N: главный антагонист; договор остаётся нижней альтернативой до конца.
- **Ника.** Сопровождает Валерию по служебным сценам трёх линий; не решает сюжет оговорками.
- **Артём.** После `c02` — живое напоминание цены договора; в линии В — добровольный свидетель (день 17); в М и N — одно эхо-упоминание максимум.
- **Наблюдатель.** Скрытый рассказчик всех линий; в В входит как нейтральный Ревизор без раскрытия тождества.

Правило тени: **у каждого ключевого персонажа в чужой ведущей линии есть минимум одно запланированное касание** — место в сетке, где его внесюжетная жизнь видна. Касание назначается на карте дня до прозы и проверяется на ревью.

## 5. Переменные и resolver

### 5.1 Пространство переменных (первичная редакция; реестр дополняется в фазе 2 по чтению кода)

| Переменная | Смысл | Растёт от |
|---|---|---|
| `bond_mir` | Романтическая связь с Мираэль | общая неделя + дни 8–12 (М-приоритет) |
| `bond_val` | Романтическая связь с Валерией | вечерние контакты, В-приоритет |
| `n_prep` | Осознанная нейтральная подготовка | вопросы о душе, отказ от обязательств владения, N-приоритет |
| `oblig_*` | Принятые обязательства (флаги) | конкретные поступки дней 8–12 |
| `priority_1` | Занятие дня 8: `mir / val / neut` | выбор в конце `c06` |
| `alex_*` (наследие) | Рабочие счётчики C00–C03 | не ломать, не переименовывать |

Счётчики растут только поступками и принятыми обязательствами, не фразами настроения. До конца общей недели существуют только возможности романтики; удвоенные и тройные романтические накопления запрещены.

### 5.2 Логика resolver (после дня 12)

```text
1. Если n_prep ≥ θ_n и целенаправленный поиск — ведущая линия N.
2. Иначе сверяем bond_mir с θ_m и bond_val с θ_v.
3. Пройден один порог — соответствующая линия.
4. Пройдены оба — побеждает более сильная последовательная траектория
   (учитываются обязательства oblig_*, не одиночные события).
5. Точное равенство разрешает priority_1.
6. Ни один порог не пройден — N по остаточному принципу.
```

Числа θ назначаются после карт дней 8–12 и проходят автотест на профилях игроков (по образцу `scripts/test_valeria_endings.py`, цель — 64+ профиля).

### 5.3 После resolver

Качество поступков влияет только на исход внутри ведущей линии:

- избегание в линии Мираэль → E02, а не N;
- провал равенства с Валерией → E04/E05;
- провал нейтральной практики → нейтральные нехорошие исходы.

Поздний «побег в чужой романтический маршрут» запрещён сеткой: подходящих событий для чужого романа после дня 12 не существует.

## 6. Техконтракт файлов (Ren'Py)

- Существующие `c00.rpy–c06.rpy` сохраняют имена (день = индекс + 1).
- Новые файлы именуются днём первично: `d08_m.rpy`, `d09.rpy`, `d12.rpy`, `d13_v.rpy` … `d20_n.rpy`.
- Финалы: существующие `e01–e05.rpy` сохраняют имена; новые нейтральные — `e06.rpy+` с постфиксом `_n`.
- Диспетчер `script.rpy`: `c00 → … → c06 → d08_<priority> → d09 → d12 → resolver → d13_<route> → … → d20_<route> → утро d21 → e*`.
- Общие мировые сцены дней 13–21 (`[О]/[В]`) живут в `world13_21.rpy`, вызываются из дневных файлов через `call`; маршрутный день не копирует их тела.
- Все ссылки на ассеты — через реестр `definitions.rpy`; файл без регистрации ассета не коммитится.
- Каждая новая глава проходит: посценовая карта → черновик → `scripts/style_guard.py` → ручной проход на ясность → показ автору. Объём главы 7 000–10 000 видимых слов, контроль — `scripts/chapter_gate.py <id>`.

## 8. Производственный порядок

1. **Фаза 1 — общая неделя до целого костяка:** закончить `c04` (уже в производстве, бюджет 7 900, девять сцен), затем `c05`, `c06` по утверждённым картам.
2. **Фаза 2 — дни 8–12:** объективный календарь событий назначается и вписывается в §3; посценовые карты трёх версий дня 8 и дней 9–12; каркас диспетчера и resolver с автотестом профилей; реестр переменных доводится по коду.
3. **Фаза 3 — дни 13–21:** посцение карты трёх линий, добыча банка, проза по протоколу.
4. **Фаза 4 — сверка:** тесты прохождений всех траекторий, сквозные чтения, полировка; только после этого — проектирование Истинного пути.

Правило показа: ни один новый художественный текст не показывается до прохождения блокеров (`HARD-BLOCKERS.md`, `style_guard.py`, ручная проверка ясности).

## 9. Опора канона

Обещания истории и жёсткие правила мира (`story-structure.md` §4 и §11) обязательны для каждой карты и каждой главы: авария обычна, смерть законна, водитель не злодей, Александр не избран, Мираэль не награда, Валерия не оправдывается задним числом, Наблюдатель не решает, ни один финал не даёт бессмертия.

Открытые поля (название города, титулы и т. п.) не закрепляются без необходимости сюжета — см. `story-structure.md` §12.
