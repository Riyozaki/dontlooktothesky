# Главная рабочая инструкция

**Статус:** единственный производственный регламент проекта.
**Приоритет:** последнее прямое решение автора → этот файл → канон мира и персонажей → карты маршрутов → старые черновики.

## 1. Главный принцип

Готовый результат — это не первый написанный текст. Готовый результат должен одновременно выдерживать:

- чтение без плана;
- причинность и физическую непрерывность;
- голоса персонажей;
- проектные блокеры;
- проверку Ren’Py;
- решение автора в точках, меняющих сюжет.

Если проблема относится к сцене или главе, переписывается сцена или глава. Бесконечная косметическая замена отдельных фраз запрещена.

## 2. Что нельзя решать без автора

До художественного текста обсуждаются и утверждаются:

- контракт маршрута, его жанр и главный вопрос;
- внешний конфликт и сквозной двигатель;
- роли персонажей и пределы их знания;
- романтическая дуга;
- устройство и цена финалов;
- новые правила мира;
- поступки, необратимо меняющие канон;
- посценовая карта главы.

Ассистент не имеет права помечать собственное предложение как «утверждённое автором». Утверждение существует только после прямого ответа автора.

Без отдельного согласования можно исправлять опечатки, технические ошибки, ссылки, проверки, структуру файлов и явные противоречия уже принятому решению.

## 3. Источники правды

1. Последние прямые решения автора.
2. `docs/state/project-state.md` — текущий этап и границы работы.
3. `docs/story/story-structure.md` и активный документ маршрута.
4. `docs/bible/character-bible.md` — характеры, знания и отношения.
5. `docs/bible/cosmology-bible.md` — устройство души, измерений и посмертия.
6. `SKILL.md` — авторский голос.
7. `HARD-BLOCKERS.md` — абсолютные запреты.
8. `examples.md` — калибровка на реальных авторских правках.
9. `game/*.rpy` — единственный источник канонической прозы.
10. `docs/manuscript/*.md` — автоматически создаваемые зеркала для чтения.

Историческое ревью не переопределяет текущий текст и новые решения автора.

## 4. Производственный цикл маршрута

### Этап A — обсуждение

До глав определяются:

```text
обещание маршрута
главный конфликт
цели сторон
сквозное давление
изменение отношений
цена победы
плохой исход
роль второстепенных персонажей
```

Результат этапа — короткий маршрутный контракт с явно отмеченными утверждёнными и открытыми решениями.

### Этап B — карта маршрута

Только после утверждения контракта создаётся крупная карта глав. Она фиксирует причинную цепочку, но не содержит готовых художественных реплик.

### Этап C — карта одной главы

Перед началом каждой новой крупной главы ассистент обязательно задаёт автору несколько уточняющих вопросов через инструмент `ask_user`, даже если общий план маршрута уже утверждён. Ответы фиксируют спорные решения до художественного текста.

Для каждой сцены фиксируются:

```text
POV, время и место
цель каждого участника
препятствие
скрытая информация
физическое действие
выбор или отказ
необратимое изменение
переход в следующую сцену
```

Сначала утверждается карта одной главы. Следующая глава не проектируется подробно, пока текущая не доказала свою работоспособность.

### Этап D — голосовая проба

До прозы создаются короткие реплики ключевых участников в конкретной ситуации. Проверяется, можно ли узнать говорящих без имён и служебных тегов.

### Этап E — тестовый фрагмент

Пишется 700–1000 слов из центральной сцены главы. Фрагмент проходит блокеры и показывается автору. Полная глава начинается после обратной связи.

### Этап F — черновик отдельными частями

Сцены пишутся отдельными частями внутри утверждённой карты. Во время первого черновика важнее движение сцены, тело, пространство и цели участников. Красивые фразы ничего не доказывают.

После завершения частей глава объединяется один раз. Дальнейшая работа ведётся с цельным каноническим файлом, а промежуточные версии не размножаются в активном проекте.

### Этап G — одно тройное ревью

После объединения глава один раз полностью читается в трёх художественных режимах:

1. **Читатель:** где интересно, где внимание выпало, что осталось в памяти.
2. **Писатель:** живёт ли сцена сама или обслуживает план.
3. **Редактор:** причинность, голоса, факты, тело, предметы и ритм.

Единый отчёт сохраняется в `temp/`. По нему выполняется один цельный редакторский проход в `game/*.rpy`, затем глава перечитывается для проверки результата.

Блокеры, метатекст, стилевые фильтры и технические ошибки проверяются отдельным гейтом после редакторского прохода. Они не создают дополнительный бесконечный цикл художественной полировки. Повторное полное ревью проводится только по прямому решению автора либо если правка открыла структурную поломку.

### Этап H — фиксация

После художественного одобрения:

1. сгенерировать Markdown-зеркало;
2. проверить перенос;
3. запустить статическую проверку;
4. запустить тесты логики маршрута;
5. выполнить Ren’Py lint и compile;
6. при доступном GUI проверить запуск, выборы, сохранения и rollback;
7. обновить `project-state.md`.

## 5. Критерии сцены

Сцена имеет право остаться, если:

- у участника есть непосредственная цель;
- препятствие вынуждает действовать;
- разговор меняет позицию или возможность;
- пространство можно представить;
- тело ограничивает действие правдоподобно;
- введённые предметы не исчезают без причины;
- персонаж знает только доступные ему факты;
- финал не пересказывает мораль сцены.

Передача информации сама по себе сценой не является.

## 6. Голоса

Голос проверяется через цель, словарь и защиту персонажа, а не через набор слов-паразитов.

Обязательный тест:

1. выписать реплики без имён;
2. распределить их по персонажам;
3. проверить, реагирует ли каждая реплика на предыдущую;
4. заменить реплики, которые можно без потери отдать другому герою.

Ни один персонаж не должен становиться голосом редактора, который объясняет Александру тему маршрута.

## 7. Объём

Объём возникает только через действие, сопротивление, последствие, бытовую задачу или изменение отношений.

Не считаются полезным расширением:

- повтор уже понятой мысли;
- новая лекция о том же правиле;
- атмосфера без действия;
- ещё один спор с тем же результатом;
- лор, который ничего не меняет;
- длинный переход к заранее известной точке.

Число слов — диагностический показатель, а не разрешение на наполнитель.

Для полноценных маршрутов автор установил романную главу из нескольких самостоятельных сцен с действиями и последствиями. Для нейтрального маршрута рабочий диапазон составляет 7 000–10 000 видимых слов на основную главу. Число не разрешает наполнитель; отклонение сначала проверяется по потерянному драматическому материалу.

## 8. Технический минимум

Обязательные команды:

```bash
python3 scripts/style_guard.py game/<chapter>.rpy
python3 scripts/validate_renpy_static.py
python3 scripts/test_mirael_endings.py
python3 scripts/test_valeria_endings.py
python3 scripts/audit_text_transfer.py
```

При установленном Ren’Py SDK:

```bash
RENPY_SDK=/path/to/renpy-sdk scripts/run_renpy_checks.sh
```

Зелёный скрипт не доказывает художественную готовность. Самописная проверка не заменяет настоящий Ren’Py lint.

## 9. Git и завершение проекта

- Работа ведётся только в закреплённой Arena-ветке.
- Ассистент не выполняет merge в `main`.
- Слияние делает автор после полного завершения совместной работы.
- Плохие черновики не складируются в проекте: при необходимости они доступны в истории Git.
- Markdown-зеркала не редактируются вручную.

## 10. Условие остановки

Глава готова к фиксации, когда:

- автор одобрил сюжетные решения;
- финальное чтение не требует пересборки;
- голоса различимы;
- нет нарушений канона и блокеров;
- у каждой сцены есть изменение положения;
- технические проверки проходят.

Если остаётся сомнение, текст называется черновиком. Статус «готово» нельзя использовать как форму самооценки ассистента.
